# الملحق العربي لدليل هيكلة مساحة العمل (Arabic Addendum)

## ملخص تنفيذي (Executive Summary)

هذا الملحق يشرح نفس نظام الهيكلة ولكن باللغة العربية بالكامل، بحيث يكون سهل القراءة للبشر وسهل التطبيق عند بدء أي مشروع جديد.

الفكرة الأساسية:

التفكير المشترك يبقى عالميًا، والتنفيذ يبقى داخل كل مشروع.

## مرجعية التعليمات وترتيب الأولوية (Precedence)

1. ملف الأساس العالمي:
   .github/CLAUDE skool Ai-automation society.md
2. ملف القواعد الإضافية على مستوى مساحة العمل:
   .github/AGENTS.md
3. ملفات المشروع نفسه:
   projects/<project-name>/docs و projects/<project-name>/workflows

إذا حصل تعارض، المرجع الأساسي + طلب المستخدم الصريح هما الأولوية.

## هيكلة مساحة العمل الأساسية (Workspace Layout)

- .github/ : تعليمات الوكيل المشتركة والمهارات القابلة لإعادة الاستخدام
- projects/ : جميع المشاريع، كل مشروع داخل مجلد مستقل
- shared/ : أدوات وقوالب مشتركة بين المشاريع
- .tmp/ : ملفات مؤقتة على مستوى المساحة
- .env : متغيرات البيئة المشتركة فقط إذا كانت فعلًا عابرة للمشاريع

## معيار تسمية المشاريع (Project Naming)

أي مشروع جديد يجب أن يكون داخل:

projects/<project-name>/

أمثلة:
- projects/ziyada-system/
- projects/new-client-growth/

## القالب القياسي داخل كل مشروع (Project Template)

كل مشروع يجب أن يحتوي على:

- scripts/ : سكربتات Python التنفيذية
- tools/ : سكربتات shell لتشغيل خطوات محددة وثابتة
- workflows/ : إجراءات العمل (SOP) خطوة بخطوة
- docs/ : توثيق المشروع، القرارات، المتابعات
- assets/ : ملفات مرجعية وتصدير (pdf/png/html/svg/zip)
- notebooks/ : دفاتر تحليل وتجارب
- outputs/ : مخرجات التشغيل والسجلات
- .tmp/ : ملفات مؤقتة يمكن حذفها
- app/ : كود التطبيق (واجهة/خلفية/وظائف)

## مسؤولية كل مجلد (Folder Responsibility)

### scripts/

أي سكربت Python يقوم بأتمتة، استدعاء API، معالجة بيانات، أو تنفيذ مهمة عملية.

### tools/

سكربتات shell لتجميع أو تشغيل أوامر متسلسلة بطريقة ثابتة.

### workflows/

ملفات إرشادية تشرح الهدف، المدخلات، الخطوات، الحالات الخاصة، والمخرجات المتوقعة.

### docs/

توثيق مخصص للبشر: خطط التنفيذ، تتبع العمل، ملاحظات هندسية، وقوائم فحص.

### assets/

الملفات الثابتة والمرجعية والتصديرات النهائية.

### outputs/

نتائج التشغيل والسجلات مثل logs ونتائج التجارب.

### app/

كود المنتج أو التطبيق الفعلي.

## المهارات المشتركة لكل المشاريع (Shared Skills)

الموقع الأساسي للمهارات:
- .github/skills/

سجل المهارات:
- .github/skills/REGISTRY.md

المهارات النشطة حاليًا:
- project-bootstrap
- google-workspace-oauth-setup

## طريقة إضافة مهارة جديدة (Add New Skill)

1. أنشئ مجلد جديد:
   .github/skills/<skill-name>/
2. أضف ملف:
   .github/skills/<skill-name>/SKILL.md
3. ضع YAML frontmatter مع وصف واضح لمتى تستخدم المهارة
4. أضفها في:
   .github/skills/REGISTRY.md
5. اجعل المهارة عامة وقابلة لإعادة الاستخدام
6. اجعل التفاصيل الخاصة بكل مشروع داخل docs/ أو workflows/ لذلك المشروع

## إنشاء مشروع جديد بسرعة (Bootstrap)

شغّل الأمر:

shared/tools/new_project.sh <project-name>

هذا الأمر ينشئ الهيكل القياسي للمشروع مع README مبدئي تلقائيًا.

## قواعد البحث في Explorer (Search Scope)

الافتراضي: ابحث داخل مشروع واحد فقط:

projects/<project-name>/**

لا تبحث في كامل المساحة إلا عند الحاجة.

## الأمان وإدارة الاعتمادات (Security)

1. ضع credentials و token داخل المشروع نفسه قدر الإمكان.
2. لا ترفع أي أسرار أو tokens إلى git.
3. استخدم .env المشترك فقط إذا كانت القيم عابرة للمشاريع فعلًا.
4. عند تغيير الصلاحيات (scopes)، أعد توليد token واختبر مرة أخرى.

## قائمة بدء التشغيل للبشر (Onboarding Checklist)

1. افتح المشروع الصحيح داخل projects/<project-name>/
2. اقرأ workflows قبل التنفيذ
3. راجع docs لفهم آخر حالة
4. نفّذ السكربتات من جذر المشروع
5. احفظ السجلات داخل outputs/logs
6. حدّث docs بعد أي تغيير في العملية

## قائمة صيانة شهرية (Monthly Maintenance)

1. تأكد عدم وجود ملفات تنفيذ مشاريع في جذر مساحة العمل
2. تأكد من تجاهل ملفات الأسرار والـ tokens عبر .gitignore
3. تأكد أن كل مشروع يحتوي workflows و docs
4. تأكد أن سجل المهارات محدث
5. أرشف المخرجات القديمة عند الحاجة
